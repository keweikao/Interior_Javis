# Q-Check — 規模化風險評估

> **版本**: 1.0
> **日期**: 2026-03-24

---

## 摘要

本文件評估 Q-Check 從 Phase 1（內部工具）邁向 Phase 2-3（SaaS / 行業平台）時，面臨的兩個結構性風險。這兩個風險相互關聯，共同指向一個核心問題：**目前規劃的商業模式（SaaS 月費訂閱）可能需要根本性的重新思考。**

---

## 風險一：報價數據信任問題

### 問題描述

Phase 2 計劃向其他設計公司收取 NT$3,000-8,000/月的 SaaS 訂閱費。Phase 3 更計劃建立「行業 OS」做市場定價數據和承包商媒合。但這代表設計公司必須將報價數據交給第三方平台。

**報價單是設計公司的核心商業機密**，包含：

- 每個工項的單價 → 反映公司的成本結構和利潤率
- 對不同工種的定價能力 → 暴露供應商/工班關係
- 哪裡高毛利、哪裡薄利 → 競爭策略的直接體現

讓第三方平台掌握這些數據，等同於將商業底牌攤開。若競爭對手也使用同一平台，平台方理論上可看到整個市場的定價分佈。

### 風險等級

**致命（Phase 2 的客戶獲取將直接受阻）**

### 目前架構的現狀

| 階段 | 數據位置 | 風險程度 |
|------|---------|---------|
| MVP（目前） | 瀏覽器 localStorage，完全本地 | 無風險（但只是因為還沒做後端） |
| 計劃中的 Phase 2 | PostgreSQL 中心化資料庫，userId 隔離 | **高風險**：平台方可讀取所有公司的報價明細 |
| 計劃中的 Phase 3 | 同上 + 跨公司數據聚合分析 | **極高風險**：數據飛輪需求與隱私保護直接衝突 |

### 緩解策略選項

| 策略 | 做法 | 優勢 | 代價 |
|------|------|------|------|
| **A. 本地優先架構** | 報價數據永遠不離開客戶端；雲端只同步規則與模板 | 最大化信任，消除數據疑慮 | 無法做 Phase 3 跨公司數據分析 |
| **B. 端到端加密** | 數據加密後存雲端，平台方無法讀取明文 | 保有雲端同步的便利性 | 技術複雜度高；仍無法做跨公司分析 |
| **C. 匿名化聚合** | 只上傳去識別化的統計數據（如「老屋翻新水電平均單價區間」） | 可做市場洞察但不暴露個別公司 | 需要足夠多用戶才有統計意義 |
| **D. 分離關注點** | 核心賣規則引擎 + 模板（不碰價格）；價格數據留在用戶端 | 產品定位清晰：「我們查完整性，不看你的價格」 | Phase 3 的數據飛輪需重新設計 |

### 建議

**短期（Phase 2）採用策略 D**：明確承諾「我們幫你檢查報價單的完整性，但我們不存取你的價格」。技術上，規則引擎只需要工項類別和結構，不需要單價。

**長期（Phase 3）若要做市場數據，採用策略 C**：opt-in + 匿名化聚合，而非預設上傳。

## 風險二：AI 時代訂閱制的脆弱性

### 問題描述

Q-Check 的付費價值建立在：

1. 25 條風險檢測規則
2. 60+ 個工項模板（含「包含/不包含」）
3. 結構化的報價編輯 UI

但在 2026 年的現實中，一個設計師可以：

- 把現有的報價單（Excel 或 PDF）餵給 ChatGPT / Claude
- 附上幾段提示詞描述台灣室內裝修的常見漏項
- **免費得到 80% 相同的檢查效果**

Q-Check 的 `PRIOR_KNOWLEDGE.md` 已經將核心知識整理成結構化文字，這些知識本身就是一個高品質的 prompt。

### 為什麼 SaaS 訂閱正在被侵蝕

| 傳統 SaaS 護城河 | AI 時代的現實 |
|---|---|
| 功能複雜度 → 用戶被鎖定 | AI 降低了「自己做」的門檻 |
| 數據累積 → 網絡效應 | 但用戶越來越不願意交出數據（見風險一） |
| 每月穩定付費的心理慣性 | 用戶開始質疑「我一個月只用 3 次，為什麼付月費？」 |
| 專業知識封裝在軟體裡 | 同樣的知識可以封裝在一段 prompt 裡 |

### Q-Check 特別脆弱的地方

目前的規則引擎是確定性的 if-then 邏輯：

```
如果有「拆除工程」→ 檢查是否有「垃圾清運」
如果建築屋齡 > 30 年 → 檢查是否有「水電管線更新」
```

這類規則，任何 LLM 都能執行，甚至更靈活（能處理模板未覆蓋的邊界情況）。用戶沒有理由每月付費給一個寫死 25 條規則的系統。

### Q-Check 有哪些是 AI 不容易取代的

1. **結構化的編輯體驗** — AI 能給建議，但無法提供一個順手的報價編輯 UI
2. **流程嵌入** — 嵌入日常工作流（打開 → 編輯 → 檢查 → 輸出 PDF）比每次開 ChatGPT 貼報價單方便
3. **Vivian 的持續策展** — 持續根據實務案例更新規則和模板，這種判斷力是 AI 無法自行產生的

### 風險等級

**高（直接影響 Phase 2 的付費轉換率和留存率）**

### 替代商業模式選項

| 模式 | 做法 | 優勢 | 風險 |
|------|------|------|------|
| **A. 按次計費** | 每次檢查報價單收 NT$50-100 | 降低採用門檻；用戶按價值付費 | 收入不穩定；需大量用戶 |
| **B. 免費工具 + 賣知識** | 編輯器免費開放；賣進階模板包、線上課程、案例分析 | 工具快速擴散；知識才是真正差異化 | 需要建立內容產出的持續能力 |
| **C. AI Agent 模式** | 不賣「系統」，賣一個能直接讀 Excel 報價單的 AI agent | 符合 AI 時代用戶習慣；不強迫換工具 | 技術門檻高；需要強大的 prompt engineering |
| **D. 顧問 + 工具** | 工具是獲客手段，真正收費的是 Vivian 的顧問 / 培訓服務 | 人的判斷力最難被 AI 取代 | 不易規模化；Vivian 的時間有限 |
| **E. 混合模式** | 免費基礎版（本地規則引擎）+ 付費進階版（AI 加強檢查 + Vivian 策展的模板更新訂閱） | 分層滿足不同需求 | 需清楚定義免費/付費邊界 |

## 兩個風險的交互影響

這兩個風險不是獨立的，而是相互強化的：

```
報價數據是商業機密        AI 能免費做 80% 的檢查
        ↓                         ↓
 不願意上傳到雲端     ───→   更沒理由付月費給 SaaS
        ↓                         ↓
 Phase 2 獲客困難            付費轉換率低
        ↓                         ↓
        └──────→  Phase 3 數據飛輪無法啟動  ←──────┘
```

### 核心結論

> **Q-Check 的長期價值可能不在「軟體訂閱」，而在「行業知識的持續策展」。**

軟體是載體，但不是護城河。Vivian 10 年經驗提煉出的判斷力——什麼該查、什麼不用查、什麼情況下例外是合理的——才是真正稀缺的。問題是如何將這個稀缺性變現，而不是把它鎖在一個月費制的 SaaS 裡。
