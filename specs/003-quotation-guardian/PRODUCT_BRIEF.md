## 一、產品定位

**設計公司的風險管理決策系統**

Q-Check 不是報價工具，是幫設計師在報價階段就避免未來施工出錯的決策系統。

**殺手句**: 「我們不是在檢查錯誤，我們是讓錯誤一開始就不會發生。」

### 產品哲學

> 「培養設計師，不是取代設計師。設計師的底氣來自知道自己準備萬無一失，不是知道自己有超屌工具。」
> — Vivian（共同創辦人/設計師）

| 原則 | 具體做法 |
|------|---------|
| 系統找風險（效率） | 即時偵測漏項、認知落差、數量異常、預算不合理 |
| 設計師懂風險（底氣） | 每次提醒都附帶「為什麼」的解釋 |
| 設計師做決定（信心） | 所有提醒都可以 Override + 加註原因 |
| 減少反覆（槓桿） | 報價階段就攔截問題，不讓錯誤留到施工階段 |

### 這個產品最有價值的不是 AI，也不是資料，是你們公司 10 年來的設計邏輯。

## 二、真痛點 vs 假痛點

### 真痛點（經過驗證）

| 排序 | 痛點 | 說明 | 後果 |
|:---:|------|------|------|
| 1 | **認知落差** | 報價寫了，但雙方對「含什麼」理解不同。例：木作櫃體，客戶以為含五金；油漆，客戶以為含全批土 | 糾紛、信任破裂 |
| 2 | **漏項** | 保護工程、清運、五金、收邊等常被遺漏 | 直接侵蝕毛利 |
| 3 | **現場條件低估** | 無電梯搬運、老屋牆面狀況未預估 | 追加費用比漏項更傷 |
| 4 | **預算不合理** | 接了預算根本做不了的案子 | 整案利潤全失，最致命 |
| 5 | **工班溝通落差** | 報價寫太模糊，工班理解不同 | 重做、延期 |

### 假痛點（不要被騙）

| 假痛點 | 為什麼是假的 |
|--------|------------|
| 報價太慢 | 報價佔案件生命週期比例小，壓縮 50% 對整體產能影響有限 |
| Excel 不好用 | 設計師已經有自己的模板，習慣了 |
| 需要 AI 自動生成報價 | 取代設計師判斷 ≠ 培養設計師 |

## 三、目標用戶

### 公司背景

- 8 人團隊（含創辦人），資深（10 年+）+ 年輕（1-3 年）混合
- 每間分公司/每個設計師 Excel 格式不同，但邏輯差不多（按工種分類）
- 年輕設計師開的報價需要資深覆核 → 佔用資深大量時間
- 高產能設計師同時跑多案（施工中 + 洽談中 + 收尾中），大量時間花在客戶溝通

### 不同角色的價值

| 角色 | 真正的痛 | Q-Check 解決什麼 |
|------|---------|-----------------|
| **老闆** | 8 人只能接 20 案，想接 40 案 | 資深覆核時間釋放，1 個資深能帶更多年輕 |
| **資深設計師** | 最耗時間的是「解釋報價」和「幫年輕收尾」 | 系統先做 70% 規則性檢查，只把 30% 經驗判斷推給資深 |
| **年輕設計師** | 不確定自己漏了什麼，要等資深空下來才敢發 | 即時風險提醒 + 學會「為什麼」，不用等人 |

### 效率槓桿：不是省時間，是減少反覆

```
錯誤發生位置：    報價階段
錯誤被發現位置：  施工階段
修復代價：        10-20x

Q-Check 的價值 = 在報價階段就攔截 → 減少施工階段的反覆修改
                → 資深設計師不用花時間「收尾」和「解釋」
                → 多出來的時間拿去經營客戶、接新案
```

## 四、系統四層架構

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 0：預算可行性快篩                                      │
│                                                             │
│  填入案件條件 + 客戶預算                                      │
│  → ⚠️「30 坪中古屋全翻 180 萬，合理區間 250-350 萬，          │
│       預算缺口 70-170 萬，建議與業主重新對齊期望」              │
│  → ✅「預算合理，可以開始報價」                                │
└──────────────────────────┬──────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Layer 1：結構化報價開立                                      │
│                                                             │
│  選案件類型（新成屋/中古屋/老屋翻新）                          │
│  → 帶入標準工項骨架 → 設計師在骨架上刪改                       │
│                                                             │
│  每個工項要求寫清楚：                                         │
│  ├─ 工項名稱 + 規格說明                                      │
│  ├─ ✅ 含：五金安裝、表面處理                                 │
│  └─ ❌ 不含：特殊五金、玻璃                                   │
│  → 解決「認知落差」#1 痛點                                    │
└──────────────────────────┬──────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Layer 2：即時風險提醒                                        │
│                                                             │
│  呈現方式：右側提示卡片（不打斷操作）+ 完成後統一報告           │
│                                                             │
│  🔴 漏項提醒（工序連動）                                      │
│  🟡 描述模糊提醒（「木作櫃體」建議補充是否含五金）              │
│  🟡 數量異常提醒                                             │
│  🟡 現場條件衝突                                             │
│  💰 單價行情提醒（低於市場行情時）                             │
│                                                             │
│  ✏️ 所有提醒都可以 Override + 加註原因                        │
│     例：「此為合作多年工班，有特殊價格」                        │
└──────────────────────────┬──────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Layer 3：完成確認 + 輸出                                     │
│                                                             │
│  📋 Checklist（設計師逐項確認）                                │
│  📄 PDF 報價單（工項描述清晰、含/不含寫明）                    │
│  📊 風險報告（內部用，列出所有 Override 和剩餘風險）            │
└─────────────────────────────────────────────────────────────┘
```

### Override 機制 — 系統的靈魂

```
系統提醒：「此單價低於市場行情」
設計師選擇：
  A) 接受建議 → 調整單價
  B) Override → 填寫原因「合作多年工班特殊價格」
     → 系統記錄，不再對同一情境重複提醒
     → 資深覆核時可以看到所有 Override 清單

沒有 Override = 限制設計師 = 會被棄用
有了 Override = 賦能設計師 = 系統知道你是故意的，不是不小心的
```

## 五、年輕設計師最常犯的錯誤（系統能防 vs 不能防）

### 可由系統防住（70%）

| 錯誤類型 | 範例 | Q-Check 怎麼防 |
|---------|------|----------------|
| 漏項 | 拆除沒報清運、浴室沒報防水 | 關卡 1：工序連動規則 |
| 工項未拆細 | 「木作」一項打天下，沒拆成天花板/隔間/櫃體 | 範本庫強制拆細 |
| 描述模糊 | 「油漆工程 一式」沒寫含不含批土 | 關卡：描述清晰度檢查 |
| 工法順序 | 先報油漆再報拆除 | 排序提醒 |
| 報價結構 | 少了保護工程、少了清潔工程 | 骨架帶入 + 缺類提醒 |

### 無法由系統防住（30% — 需要資深覆核）

| 錯誤類型 | 說明 | 為什麼系統防不了 |
|---------|------|----------------|
| 現場判斷 | 牆壁狀況、管線走法 | 需要眼睛看、手摸 |
| 客戶需求誤判 | 沒聽懂客戶真正想要什麼 | 需要溝通經驗 |
| 美感判斷 | 材質搭配、空間比例 | 設計專業 |
| **預算策略** | 這案子該不該接、怎麼配預算 | **最關鍵的經驗判斷** |

## 六、商業模式

### 短期（0-6 月）：內部工具

- 自己公司 + 合作方使用
- 量化成效：
  - 追加金額下降比例
  - 資深覆核時間減少比例
  - 報價到發包的來回修改次數

### 中期（6-18 月）：SaaS 訂閱

- 開放給市場上的設計公司
- 定價模式：月費（按公司人數）
- 核心賣點：「同樣的人接更多案子」

### 長期（18 月+）：設計產業的作業系統

- 結構化報價數據 → 同業比價、市場行情
- 串接建材商、工班派工
- AI 風險預測（Phase 3，基於累積的乾淨數據）

## 七、正確的建設路徑

```
❌ 錯誤路徑（會死在資料地獄）：
   蒐集舊報價單 → 建資料庫 → 訓練 AI → 做系統
   → 報價單格式不一，資料清理永遠做不完

✅ 正確路徑：
   Phase 1: 結構化報價系統（不做 AI）
            → 用戶使用系統自然產生乾淨資料
   Phase 2: AI 整理舊資料，擴充工項與漏項庫
   Phase 3: AI 風險預測與報價建議（基於高品質數據）
```
